Ramblings on Financial "Advice"3/3/2014

I'm in the middle of wrapping up my studies at Bitmaker Labs, and deciding how best to put all the information we've been presented on modern web development to use after this whole thing wraps up. I've long been interested in financial markets by way of my work (Manulife), and I suppose, how the 'average' person makes sense of the plethora of personal financial advice. According to CBC, almost a third of Canadians rely on professional financial advisors to guide their decisions with respect to managing debt, a mortgage, education and retirement savings. And it turns out those many of them are not that good. While there has been some headway leveraging modern technologies (web apps, mobile apps) to better inform consumers of financial products, it's my humble opinion that we still have a long way to go. So that's my starting point.

There's a database of financial professionals registered in Canada held online at Canadian Securities Administrators (CSA). It holds records, both active and historical, of 290,000 registered investment professionals, as well as their location, employer, phone, title, tenure, credentials and disciplinary action. According to their website, they are "Securities regulators from each of the 10 provinces and 3 territories in Canada have teamed up to form the Canadian Securities Administrators (CSA). The CSA protects Canadian investors from unfair, improper, or fraudulent practices and fosters fair and efficient capital markets." The database is hard to interact with, and looks ripe for some scraping and reconstitution.

A major naivite of mine entering this course was the notion that "learning to program" is the same as "programming". It's not. Put another way, there is an enormous amount of programming, software, or techniques that you are required to learn in the process of completing a project. So while Bitmaker Labs focuses largely on the Ruby language, I've come accross many a situation that demands an alternative method. Case in point, iMacros web scripting abilities are far better suited to traversing and saving this CSA database in .txt files than a ruby gem or methodology. Am I familiar with iMacros syntax, potential, and particularities? No. But I've been able to piece together just enough functionality to perform the function I need. It's good enough for me, and it should be good enough for you too.

Dictated by the current structure, the iMacro script will need to traverse 290,000 pages to scrape all available details on active/historical financial advisors. It's unfortunate, but the CSA offers no way to search for active users only. Alas. Once I've left this script running for about 28 hours (8 seconds per record, you do the math), I should have all the info I need. Then RUBY can do its thing. That is, I'll be using Ruby to parse the .txt files (but not to create them).

One of the characteristics of the .txt files being created is that each contains only one user record, and this record tends to start at line 370 (spot checking confirmation). The status (active, historical) is available on line 380 (again, spot checked). Since we are only interested in constructing a database on active financial advisors, ostensibly we can divide our text records into two groups: active and historical. Once complete, we can create a mega text document with only active users, containing all that wondeful information that we can then investigate. How many active advisors are there? How many in Pickering? How many have been disciplined? How many are male? How many are credentialed (CFA, CA, CFP)? How many work for RBC? Dundee Wealth? Exciting stuff.

Because you can write javascript in your iMacro, you can create loops where otherwise the instructions would be explicit. In my case, I need to paginate through 2,900 pages (100 records/page). This is well suited to a loop, and I am currently investigating ways to best write this. I suppose I'll keep you up to date on my progress. Don't hold your breath. You might pass out.

I'm starting out ambitiously, as I hope to cover several applications (unix, sublime text, chrome, git, mac os). So this will mostly act as a resource for me, but I hope it may provide some shortcuts that you find helpful.

First off, I hadn't heard of the command line until about 6 weeks ago. I am a child of the GUI, and having been presented with the option to point click, it's no doubt the easiest way to proceed navigating through your computers directories. However, it's a much more productive ue of time to learn these commands for use in the Command Line Interface (CLI). To launch 'Terminal' in Mac OS, hold 'command + spacebar' to bring up Spotlight. Type in 'Terminal' and hit 'enter'. This is a CLI. You can use it to rename, move, delete files, navigate directories and list their contents etc... There are a handful of core commands that represent about 90% of the tasks I typically need to complete. Mac OS uses unix, and so I will only be covering these commands.

Unix
  • cd: change directory
  • cd ..: back one directory
  • cd ../..: back two directories
  • cd ~: home directory
  • cp -r currentFolder newFolder: make a copy of folder in same directory
  • rm: remove a file
  • mkdir: make a directory
  • rmdir: remove a directory
  • mv: move a file/directory
  • pwd: present working directory
  • ls: list contents in directory
  • ls -l: list content details in directory
  • ls -a: list all contents in directory
  • touch: create a new file
  • cat: display contents of a file
  • cmd + K: clear shell
  • nano ~/.bash_profile: launch bash_profile to create an alias

Git + Github is awesome once you get the hand of it. A universally adored version control manager.

Git
  • git status: current status of working directory
  • git log: history of commits
  • git add --a: add files to be pushed to repository
  • git commit -m "message": commit files to repository
  • git push -u origin master: push files to repository
  • git fetch --all | git reset --hard origin/master hard pull from master
  • git push origin --delete branchName delete branch locally/remotely

Chrome is certainly, from a developer's perspective, one of the nicer browsers to work in. There is a simple set of 'Dev Tools' that can be launched with the following command; 'cmd + optoin + J'. You can do most anything here, like inspect the source code, or edit the style, position, and content of a particular element. For reasons I may one day be able to explain, webpages take on different 'feels' in different browsers. Already, in my work, I'm noticing that fonts I imported for my homepage only work in Chrome and Safari, but not in Firefox. This adds a layer of complexity to developing webpages that respond as desired accross different browsers. So I take the following approach. I'll typically style my page so it works in Chrome, test it in Firefox and Safari, and if there are differences, I'll curse aloud.

Chrome
  • cmd + R: refresh page
  • cmd + T: new tab
  • cmd + L: highlight address bar
  • cmd + shift + J: open downloads page
  • cmd + option + J: launch 'dev tools'
  • cmd + option + U: view source code
  • ctrl + tab: cycle through open tabs
  • option + tab: cycle through "header"s tags
  • cmd + ~: cycle through open windows

Similar to the way the CLI can make light work of navigating through your directories, keyboard shortcuts can take a tremendous amount of pain out of navigating accross and within applications. I've taken just a small sampling of a handful of shortcuts that I've found to be major timesavers. Yes, there are only 2. You will be thrilled to know them.

Mac OS

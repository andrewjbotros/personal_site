February 2, 2014

So, today's the Big Game Day (god forbid I use superbowel. God forbid I use god. I digress.) Against my better judgement, depending on how you look at it, I've decided to do the honourable thing and spend a lazy Sunday studying rails and generally getting caught up. One of the things I'm slowly trying to master is the ability to manipulate all necessary applications and functions directly from the keyboard. While it certainly takes a fair bit of up front effort, you'll find that over time, if you stay disciplined, you fingers will catch up with your brain and it will become second nature. Atleast, that's what I keep telling myself. So this post I wanted to review the handful of shortcuts that I'm trying to introduce into my workflow, and thus are the ones I need to remind myself of. I'm starting out ambitiously, as I hope to cover several applications (unix, sublime text, chrome, git, mac os). So this will mostly act as a resource for me, but I hope it may provide some shortcuts that you find helpful.

First off, I hadn't heard of the command line until about 6 weeks ago. I am a child of the GUI, and having been presented with the option to point click, it's no doubt the easiest way to proceed navigating through your computers directories. However, it's a much more productive use of ones time to learn these commands for use in the Command Line Interface (CLI). And you can use it to rename, move, delete files, navigate directories and list their contents etc... There are a handful of core commands that represent about 90% of the tasks I typically need to complete. As I'm using a Mac OS, I'll be covering only unix commands.

Unix
  • cd: change directory
  • cd ..: back one directory
  • rm: remove a file
  • rmdir: remove a directory
  • mv: move a file/directory
  • pwd: present working directory
  • ls: list contents in directory
  • ls -l: list content details in directory
  • ls -a: list all contents in directory

Git + Github is awesome once you get the hand of it. A universally adored version control manager.


Git
  • git status: current status of working directory
  • git log: history of commits
  • git add --a: add files to be pushed to repository
  • git commit -m "message": commit files to repository
  • git push -u origin master: push files to repository

Chrome is certainly, from a developer's perspective, one of the nicer browsers to work in. There is a simple set of 'Dev Tools' that can be launched with the folowing command; 'cmd + optoin + J'. You can do most anything here, like inspect the source code, or edit the style, position, and content of a particular element. For reasons I may one day be able to explain, webpages take on different 'feels' in different browsers. Already, in my work, I'm noticing that fonts I imported for my homepage only work in Chrome and Safari, but not in Firefox. This adds a layer of complexity to developing webpages that respond as desired accross different browsers. So I take the following approach. I'll typically style my page so it works in Chrome, test it in Firefox and Safari, and if there are differences, I'll curse aloud.


Chrome
  • cmd + R: refresh page
  • cmd + T: new tab
  • cmd + L: highlight address bar
  • cmd + option + J: launch 'dev tools'
  • ctrl + tab: cycle through open tabs
  • cmd + ~: cycle through open windows

Similar to the way the CLI can make light work of navigating through your directories, keyboard shortcuts can take a tremendous amount of pain out of navigating accross and within applications. I've taken just a small sampling of a handful of shortcuts that I've found to be major timesavers. Yes, there are only 2. You will be thrilled to know them.


Mac OS
  • cmd + spacebar: spotlight allows you to search for, and openany file or program
  • cmd + tab: cycle through open programs

I'm currently using something called Sublime Text 2, a very common text editor used amongst developers. Dare I include myself in this group? Hells, why not. I mean, I am using Sublime Text 2. Naturally, there are a handful of things one does in sublime that certainly make working with it a lot easier. In short, it's nothing more than a word like editor (without fonts) that makes writing a ruby, javascript or python file enjoyable. There's lots of nifty autocomplete features, ways to navigate through windows and search for files that are seriously handy.


Sublime Text 2
  • subl .: launch sublime from terminal
  • cmd + ~: cycle through open windows
  • cmd + option + #: view in # tabs
  • cmd + shift + option + #: view in # tabs horizontally

I want to see all the stock footage of Bill Gates sucking it up on seemingly simple problems. And then, through stop motion film, show how many damn hours he put into solving his problems. Did you hear that Will.I.Am is at MIT studying computer science because he's convinced you'll be a ninja with the skill set? Well, there's not a lot that me and Will.I.Am agree on. But, I certainly feel naked in this age to not know how to program even the simplest of apps. And so, the ideas keep piling up. Now, in the spirit of sharing not just sucesses, but 'failures' (lessons learned), I'll share something that presented a major challenge.

I toiled for 6 hours (because, why not) on my ever so simple website trying to center content. Yes. That's right. Centering content. You know when you're building a simple website, and you ask your developer, "Hey, can you, you know, center the content?" - you'd expect - "Heh heh... uh yeah. Duh. *arrogant clackity typing noises* DONE!" Well, if I was your developer that would have taken me 6 hours. I'd like to say I learned a lot about html. And I did. And I'd like to tell you that I will proceed to review what I learned with you. Well, I'm not. I learned that you can't center content in a div tag. You know what else I learned? I don't know how to type /div/ tag with the typical <> brackets without breaking something. I need to learn that tomorrow.

January 13, 2014

Officially day 1 in Bitmaker Labs. Very impressive space, very confident team. If you've been fortunate enough to work in the beaurocracy of a large corporation, then perhaps the first thing will strike you is how much got accomplished by this relatively small group of people. Amazingly, only 5 cohorts in (1 yearish), they will be moving to a new space twice its current size at the end of February.

There's some evidence that suggests that when recessions hit, some of those individuals that find themselves left in the wilderness learn how to hunt. Really well. Put another way, there's the old joke "How do you get a beaurocrat to reach his full potential? Fire him." With that said, it certainly feels as though most people who have made it this far (quit their job to learn development) are testing whether or not they'll be able to forage for edibles. That's the first observation. The second is that people are not learning to code for the first time. Many of tried and "failed" before, and I certainly count myself in both groups.

I often relay this example, but I was forced to take an "Intro to Computer Science", or intro to Java as a requisite to graduate, and I was terrible at it. So difficult in fact, that I proactively avoided having to learn it. The syntax was overwhelming. The classifications were confusing (objects, methods, classes). And the potential to create something that could provide some utility outside of adding to numbers, or saying "Hello" seemed out of hand. So, what prompted the change of heart?

Software is eating the world. That's not my line of course, but, I certainly do believe it. Retail, Transportation, Finance, Medicine and Law will be fundamentally transformed by employees moonlighting as programmers. Atleast, I *think* so. Boom.
